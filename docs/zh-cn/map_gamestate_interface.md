## 地图与游戏状态接口

鉴于我（Schombert）根本没有在地图方面工作，我们需要在地图与程序其余部分之间安排一个固定的接口（目前，仅针对原始地图数据本身）。特别是，我们需要决定首次加载场景时创建的持久地图数据的格式。

### 省份标识符

对于主游戏状态，省份由 `dcon::province_id` 类型标识。然而，这对地图来说可能不是最方便的。`dcon::province_id` 在内部将"不是有效省份"的状态存储为 -1 索引，这对于用作纹理坐标等来说有点尴尬。因此，我们将在 `province.hpp` 中提供两个工具函数：`uint16_t to_map_id(dcon::province_id)` 和 `dcon::province_id from_map_id(uint16_t)`。这些函数将省份 ID 类型转换为无符号整数，其中 `0` 表示"无效"省份，所有其他省份从那里开始顺序计数。要获取省份数量，你可以查询 `state.world.province_size()`。加 1 就能告诉你需要多大的数组或其他结构来容纳 `to_map_id` 可能返回的所有值。

### 地图数据

在场景构建过程的某个时刻，地图部分应提供一个可调用的函数，将源地图数据（即图像文件）加载到字节数组中。具体来说，地图"模块"应提供以下函数 `void load_map_data(sys::state& state, ankerl::unordered_dense::map<uint32_t, dcon::province_id> const& color_map, error_handler& err)`。第一个参数只是状态对象（因为你可能需要文件系统对象来打开地图图像，对吧？）。第二个参数是一个映射，给出了打包为 `uint32_t` 值的颜色值与省份 ID 类型之间的规范映射。要从颜色数据获取必要的 `uint32_t` 值，请调用 `container_types.hpp` 中提供的 `uint32_t sys::pack_color(int32_t r, int32_t g, int32_t b)` 函数（严格来说，将值打包到一个 int 中使其成为一个容器，对吧？）。你可以使用标准 `find()` 函数配合此映射，如果颜色不在映射中，它将返回一个等于 `color_map.end()` 的迭代器，如果颜色在映射中，你可以通过 `returned_iterator->second` 获取省份 ID。

地图有两种可能性。首先你应该检查路径中是否存在 `provinces.png`。如果存在，你应该记住正在加载一个"新"地图，并从中读取数据，就好像它覆盖了整个地球表面一样，即最上面一行是北极，最下面一行是南极。如果找不到该文件，你应该转而尝试加载 `provinces.bmp`。在这种情况下，你应该估算顶部和底部的纬度（不是从文件中获取；只是计算一个你在此情况下始终会回退到的估算值）。然后，使用文件的垂直大小，计算如果 `bmp` 延伸到南北极需要多少行。然后读取位图中的数据，只是在读取开始时，用 `0`（无效省份值）填充地图数据的顶部区域直到你的纬度估算值，然后读取文件数据，最后再用 `0` 填充剩余的空间直到南极。**注意**，渲染时，你应该将这些 `0` 显示为水域，并且应该直接丢弃对它们的任何 UI 点击。建议采用上述程序的原因是，在地图加载后，相同的例程可以使用地图数据，无论它来自旧地图还是新地图（尽管小地图可能需要一些特殊逻辑）。地图数据应在内部存储为连续的 uint16_t 地图 ID 值块，其尺寸和大小应存储在 `sys::state` 中的某个位置（具体位置由你决定），并且你应该在本文档中记录该信息，以便以后可以正确地序列化和反序列化。

地图加载后，你还需要地形地图。从概念上讲，这是相同的东西，只是我想你也许会希望将数据存储为 uint8_t 值数组。与省份地图一样，如果正在使用旧的地形 `bmp`，需要填充额外的数据以覆盖整个地球。另外还有一个问题，你需要决定希望"新"地图的 `terrain.png` 如何存储地形信息。有人（好吧，可能是我，我知道这种事是怎么回事）必须为这个新地图创建新的地形图像，所以尽量让这个人的工作轻松一点。如上所述，你需要决定在哪里存储在 `sys::state` 中读取的数据，并记录该选择以便可以序列化和反序列化。

**不要**在此过程中创建任何 OpenGL 资源。场景可能在 OpenGL 尚未准备好甚至没有窗口的情况下创建。地图创建过程的那部分需要自己的函数，我建议将其放在平台无关的 OpenGL 初始化函数中（即我们创建着色器的地方）。该 OpenGL 初始化函数将*始终*在场景创建或加载*之后*调用，因此届时地图数据将可供你使用。
