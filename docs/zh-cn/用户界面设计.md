## 用户界面设计文档

本文档**不**涉及地图/地图渲染。本文档**不**涉及UI系统的代码。本文档涵盖UI系统背后的通用思路/编写代码的计划。本文档也只从相对较高的层面介绍问题。具体细节可能只会出现在其实现文档中。

### 与 OpenGL 配合工作

所有 UI 元素的渲染逻辑都包含在一对片段/顶点着色器中。为了获得针对特定类型 UI 元素（例如渲染饼图而非简单纹理矩形）的特定渲染能力，可以打开或关闭其中的各种函数组合。理想情况下，这些将直接从 open v2 复制过来，然后我再也不会碰它们。

###  quirks

大多数 UI 系统的工作方式如下：每个 UI 节点（我经常将这类东西称为元素）都有一个边界矩形，然后该矩形内的任何输入和渲染工作都在该元素与其子元素之间协商。这意味着树结构允许你有效地过滤掉大部分事件，这样你就不必查看大多数元素来确定鼠标点击发生了什么。

但 Paradox 制作其 UI 系统的方式似乎并非如此。在我看来，在某些地方，元素会故意将子元素定位在自身之外。这意味着消息必须发送到所有地方，因为即使消息与父节点无关，它也可能与子节点或某些子节点的子节点相关。这很不幸，但我认为直接模仿这种 nonsense 比试图修复它更容易。Open V2 采用了更合理的方法，但这意味着我必须不断对元素应用动态的“修复”，以使子元素适应父元素，这带来了不小的额外麻烦（因为我永远无法相信 GUI 描述会按原样工作）。

### 事件与消息

事件（例如鼠标点击）会按某种顺序（通常从“最顶层”向下）传递给 UI 元素。对于某些事件，例如鼠标点击，一旦事件处理程序发出已消耗事件的信号，该过程就会立即停止。其他事件，例如鼠标移动，则仅被记录为已被某人拦截。事件通过 UI 系统传递后，任何未以某种方式处理的事件（意味着它们不“在”任何 UI 元素之上或不“针对”任何 UI 元素）将被传递给地图。

### 自定义行为

UI 系统的关键特性是让各种控件执行操作。为此，我们首先需要能够将事件处理程序附加到 UI 元素。每个 UI 元素都将包含一个 `unique_ptr`，它可能指向一个自定义处理程序。为了附加这些指针，当我们创建 UI 元素时，我们将使用其名称加上某个哈希映射来查找是否已为该类型的元素注册了全局行为。如果有，该映射将为我们提供一个用于创建新 `unique_ptr` 的函数。

### 自定义数据

大多数 UI 元素需要存储一些数据来确定它们显示什么/如何表现。这通过两种方式管理。首先，在元素创建时，当元素变为可见时，以及每当游戏状态发生变化时，更新消息会传播到可见的 UI 元素，允许它们从游戏状态中拉取数据。其次，为了避免到处复制相同的数据（例如，如果国家窗口跟踪你正在查看哪个国家，则并非每个子控件都需要复制该信息），每个控件都将被赋予能力，从概念上讲，填写一份请求某些信息的表单，该表单将沿 UI 层次结构向上传递，以查看是否有任何元素可以填写它。因此，例如，国家窗口中的控件可以简单地询问是否有任何国家存储，并从包含它们的窗口获得答案。从技术层面来看，这将通过 `std::any` 实现。一个元素将使用它想要的数据类型的默认构造类型初始化一个 `std::any`。然后这将被向上传递，直到有人能够用数据填充它。（如有必要，相同的机制也将用于外部设置，以在常规更新机制之外设置存储在控件中的数据。）

### 内存管理

我们不会进行任何内存管理。具体来说，任何 UI 元素或其指针背后的行为都不会被删除或在内存中移动。当我们希望 UI 中的某些内容消失时，我们只会隐藏它。在许多情况下，这将简单地工作，因为从概念上讲，该 UI 部分只有一个唯一实例。例如，只有一个省份窗口。如果我们确实遇到需要或多或少无限数量 UI 元素的地方（例如，我们正在制作 UI 元素以放置在视野中的每个地图省份上），我们将从池中获取这些元素，重用当前未标记为可见的元素，直到耗尽池，然后再创建新的。这些池自然会找到一个上限，在这一点上，将不再进行任何 UI 元素的创建或删除。所有这一切的重点是，主要是，它允许我们自由地存储指向 UI 元素的指针，而无需担心，因为可能发生的最坏情况是你正在与一个不可见的元素或已被重用的元素对话。
