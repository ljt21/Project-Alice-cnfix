## 游戏中的文本数据

游戏的文本数据主要来自 `localisation` 目录中的 `.csv` 文件，尽管各种其他字符数据也因缺乏更好的归宿而存放在这里。`.csv` 文件中的每一行首先由一个标识文本的标签组成，然后是若干列，每列对应一种不同语言的翻译（尽管其中许多是空的）。还有以 `#` 开头的行表示注释。请注意，这不是一个符合标准的 `.csv` 文件。引号（`"`）在 `.csv` 格式中具有特殊含义，而在这里它们只是引号（而且它们并不总是成对出现，这就是为什么遵循 `.csv` 标准会破坏你对这些文件的解析）。在文本数据中存在一些特殊之处。首先，你可能会遇到值为 `0xA7` 的字符后跟一个大写字母。这表示其后的任何文本必须以不同的颜色显示。还有成对的 `$` 字符包围着需要在显示文本之前替换的变量。最后，`\n` 序列用于表示换行。为了捕捉这些特殊之处，特定键的文本不会按原样存储。相反，它会被分解成一系列组件。每个组件要么是：一段纯文本、一次颜色变化、一个待替换的变量，或一个换行。

在某些情况下，`0xA7` 转义可能是一个 3 字节转义序列，由 `0xEF`、`0xBF` 和 `0xBD` 字符依次组成，后跟一个颜色代码。对于问号 `?` 字符还有一个特殊的处理情况；因为它们可能后跟一个颜色代码，如果确实后跟颜色代码，那么它们将被视为转义序列。
颜色代码可能是以下之一：
- `W` 白色。
- `R` 红色。
- `G` 绿色。
- `Y` 黄色。
- `b` 黑色。
- `B` 蓝色。
- `L` 浅棕色（淡紫色）。
- `O` 橙色。
- `g` 浅蓝色。
- `l` 浅灰色。
- `C` 深红色（深红）。
- `J` 深绿色（丛林绿）。
- `!` 重置颜色。
任何其他颜色代码都是未定义的。

具体来说，这些文本序列的描述存储在 `sys::state` 的 `text_sequences` 成员中。这是一个由 `dcon::text_sequence_id` 索引的容器。其中包含 `text::text_sequence` 对象，每个对象包含一个 `starting_component` 索引和一个 `component_count`。这些值可用于从 `sys::state` 的 `text_components` 成员中检索组件序列。这些组件中的每一个都是一个变体，即 `std::variant<line_break, text_color, variable_type, dcon::text_key>`。`line_break` 只是一个表示换行的空结构体，`dcon::text_key` 是字符串池中的一个索引（更多内容见下文），用作文本序列的句柄。`text_color` 和 `variable_type` 都是枚举，分别对应可能的文本颜色变化和变量类型。（在序列中遇到 `text_color` 意味着你应该切换到以该颜色渲染后续文本。）

最后，你可以通过其键*全部小写*在 `key_to_text_sequence` 映射中查找这些文本序列。在内部，此映射以 `dcon::text_key` 作为其键的表示，但你可以使用 `std::string` 或 `std::string_view` 调用 `find` 来查找文本序列。

### 字符串池

状态对象包含一个名为 `text_data` 的 `std::vector<char>`，其中包含 win1250 代码页字符值。它包含了我们可能需要提取的所有文本的汇总，这些文本是从游戏文件中提取的。**不要**在游戏运行时更改 `text_data` 的内容；只应在创建新场景或加载现有场景时向其添加数据。字符串池中的数据应通过 `dcon::text_key` 结构访问，它本质上充当一个更小的字符串视图（可以自由复制，大小为 4 字节）。有三个便捷函数可帮助处理字符串池：

- `std::string_view to_string_view(dcon::text_key tag) const` -- 此函数接受一个 `text_tag` 并将其转换为传统的 string_view。如果 `dcon::text_key` 是特殊的无效标签，你将只得到一个空的字符串视图。
- `dcon::text_key add_to_pool(std::string_view text)` -- 此函数接受 string_view 中的文本，将其附加到 `text_data` 向量的末尾，然后返回一个表示存储文本的 `text_tag`。
- `dcon::text_key add_to_pool(std::string const& text)` -- 此函数接受 string_view 中的文本，将其附加到 `text_data` 向量的末尾，然后返回一个表示存储文本的 `text_tag`。如果你有一个非空终止的字符串想添加，请使用上一个函数，但如果它已经在 `std::string` 中，请使用此函数。
- `dcon::text_key add_unique_to_pool(std::string const& text)` -- 此函数的作用与上面的函数相同，只是它首先搜索现有存储的数据，以查看该字符串是否已存储在其中。如果找到了该字符串，则不会向 `text_data` 附加新数据。否则，它的行为与上面的函数相同。如果你知道你想存储的文本已经有很大概率存在于 `text_data` 中的某个地方，此函数很有用。但是，扫描整个存储文本并非没有代价，尤其是随着文本量的增长，因此不要将此函数作为添加文本的默认方式。

实现说明：我在是否将字符串的大小存储在其标识标签（当前为 `dcon::text_key`）中，还是在 `text_data` 中存储空字符以确定一个字符串的结束位置和另一个字符串的开始位置之间反复犹豫。目前我正在存储空字符。

## 语言系统

默认情况下，Alice 将定义以下语言，对应于 Victoria 2 中使用的默认语言：
* en-US: 英语（美国）
* fr-FR: 法语
* de-DE: 德语
* pl-PL: 波兰语
* es-ES: 西班牙语
* it-IT: 意大利语
* sv-SV: 瑞典语
* cs-CZ: 捷克语
* hu-HU: 匈牙利语
* nl-NL: 荷兰语
* po-PO: 葡萄牙语
* ru-RU: 俄语
* fi-FI: 芬兰语

其中一些语言 unfortunately 不适合 Windows-1252（最著名的是俄语——即使它被定义了）。

为了克服这些限制而不破坏模组兼容性，解决方案是拥有一个语言清单 `languages.txt`，其中相应地指定了每个**附加**语言。

```
ar-AR;utf8_linked_rtl
hn-IN;utf8_linked
<...>
```

语法如下：`iso-code;setting`。ISO 代码由语言的两字符代码加上地区或国家的两字符代码组成。例如，瑞士德语可能与德国德语有所不同。

* `utf8_linked`: 连字脚本，使用 UTF8。
* `utf8_linked_rtl`: 具有 RTL（从右到左）属性的连字脚本，使用 UTF8。
* `utf8`: 通用 UTF8 脚本。
* `arabic`: 使用阿拉伯字体集，设置为连字和 RTL 脚本，使用 UTF8。
* `chinese`: 使用中文字体集，使用 UTF8。
* `japan`: 使用日文字体集，使用 UTF8。
* `korean`: 使用韩文字体集，使用 UTF8。
* `cyrillic`: 使用西里尔字体集，使用 UTF8。
* `latin`: 通用 UTF8 拉丁脚本。

子语言或方言将从父语言获取数据，例如，`cs-CZ` 将从 `cs` 文件夹和 `cs-CZ` 文件夹获取数据，首先是父语言，然后是方言版本——这是为了简化某些语言之间共享翻译的工作，否则这些语言将需要太多重复。

任何在非英语语言中未定义的本地化键，都将默认使用英语版本——为了方便起见。此外，任何需要使用回退（fallback）的语言（例如 `ALD_democracy` 回退到 `ALD`）将执行以下操作：
- 我们的语言特定本地化中是否定义了 `ALD_democracy`？
- 是？使用它。
- 否？我们的语言特定本地化中是否定义了 `ALD`？
- 是？使用它。
- 否？回退到英语中的 `ALD` 本地化。
