## 解析器生成器：工作原理

在讨论解析器生成器时，有一件重要的事情需要记住，那就是生成器（它解析一个描述我们想要从它获得的输出的文件）与它生成的解析器（可以解析游戏中附带的大多数数据文件并将其转换为有用的数据）之间的区别。因此，当我们谈论“解析器”时，请务必弄清楚我们是在谈论生成器内部的解析器，还是它生成的解析器。

### 简短的历史课

生成器及其文件格式中存在许多怪异和令人困惑的地方，这是由于其相对复杂的历史。最终如果能把它和它消耗的文件变成某种整洁干净的东西就好了，但我们现在所拥有的东西是从 open v2 项目中获取的，只做了 minor adjustments。最初，open v2 中的解析由模板黑魔法处理，它将本身结构化为类型树的类型通过使用该树每个节点上的编译时信息转换为解析器。这当时听起来极其聪明，但真是一个噩梦，而且有时会崩溃编译器。作为一个 minor improvement，解析器随后在大量宏的帮助下制作，这些宏仍然很难编译，但至少更容易使用一些。最终我意识到我可以获得一些明显的性能改进，同时大大缩短编译时间，因此我开始转向类似于我们这里的生成器。然而，这个过渡在 open v2 中并没有完全完成，而且大多数宏定义仍然存在于那里。

在 open v2 的开发过程中，随着我处理了越来越多的文件，这使我接触了各种边缘情况，并且随着我简单地尝试不同的方法以试图找到更好的解决方案，解析器的使用方式也发生了变化。结果是，宏系统 grew to incorporate 了许多不同的做事方式，为了使与该系统的兼容性更容易，解析器生成器被构建为基本上能做所有事情。我希望最终我们能够简化事情，因为我打算或多或少地从头重新创建解析器定义，我将尝试坚持一种做事方式，允许某人（我希望是其他人）回去，剥离未使用的部分，留下一些整洁干净的东西。（而且我知道一个额外的、未实现的、针对生成解析器的优化，我希望有一天能把它加进去。）

作为一个非常粗略的近似，我们可以假装生成器旨在支持两种不同的解析方法。在第一种方法中，我们通过创建一个存储组中包含的任何信息的对象，然后返回它来解析文件中的特定“组”（看起来像 `key = { ... }` 的东西）。这是一种非常自然的方式，因为它创建了一个反映文件内容的 C++ 对象层次结构，然后你可以在完成文件后轻松处理它。第二种解析方法是当你读取文件时立即将数据放入其最终目的地。为此，当你在文件中的嵌套“组”中前进时，你构建一个“上下文”——大致上是你在文件中的位置的表示，以及你需要存储数据的外部对象的引用——它会被传递给解析器，解析器又将其传递给任何它调用的用于处理数据的函数。在第一种方法中，解析器主要通过将遇到的值赋给它所关联的类的成员变量来存储值（更多内容见下文）；在第二种方法中，它主要在遇到数据时对其调用函数。

### 最终要解析的 paradox 格式文件的内容

在我编写这些各种解析器迭代时不小心发展出了一种有点特殊的术语，现在我想你必须学习它才能让理解解析器生成器更容易一点。抱歉。

在最一般的层面上，一个 paradox 格式文件是一个项目列表，每个项目可以是四种东西之一。

首先，一个项目可以是 `value` 或 `association`（抱歉，两个术语在某些地方都会使用），通常形式为 `yyy = zzzz`。`yyy` 和 `zzzz` 可以是任何不包括空格、制表符、换行符或括号的字符序列，或者它们可以是包含其中任何字符（换行符除外）的字符序列，只要它被引号包围。等号本身被称为其 `association type`。对于任何正常的 paradox 格式文件，它总是 `=`。然而，为了让模组制作更容易，我们还支持 `<`、`>`、`>=`、`<=`、`!=`、`==` 和 `<>` 作为 association type。（这不是一个新功能；它在 open v2 中已经存在，而且此时要移除它需要花费更多精力。）

下一个最常见的项目是 `group`，形式为 `yyy = { ... }`，其中 `...` 本身是一个包含零个或多个项目的列表。严格来说，任何生成的解析器都会解析单个组（我们将文件的全部内容视为一个隐式组）。当解析器遇到一个组时，它会将该组的内容交给另一个解析器，然后对返回的值执行某些操作。

然后我们有了更罕见的项目。偶尔我们会发现一个 `free value`，即看起来像只是 `yyy` 但没有后跟支持的 association type 的东西（最常见的是数字）。所有这些都发送到同一个 free value 处理程序。然后是极其罕见的 `free group`，看起来像 `{ ... }`，即一个项目列表，但没有你期望在组上找到的标签。每个列表的内容都由 free group 处理程序解析。（注意，这些东西曾经被称为 free sets 或仅仅是 sets，你仍然可能在源文件中看到一些这种术语。）

请注意，注释不需要考虑；它们在解析器看到之前就被过滤掉了。（没错，有一个微小的预处理器。）

### 为组的内容定义解析器

请记住，每个文件都是一个隐式组，因此每种文件类型的解析器都以其中之一开始，然后调用其他组解析器。

对于我们希望定义的每个解析器，我们还必须定义一个唯一的类。类的内容将用于存储解析后的数据或提供处理它的函数。这些类有两个限制。首先，它们必须是默认可构造的。其次，它们必须支持一个 `finalize` 函数，该函数接受单个参数，即馈入解析器的上下文（当解析器完成读取组时调用此函数，当你想立即对数据执行某些操作但直到你拥有该组的全部内容之前无法执行时，这很有用）。如果你的解析器不使用上下文，我建议将此参数的类型设为 `int32_t`（也许有一天我们会得到常规的 void）。

我们通过将关联的类型名称单独放在一行上，前面没有任何空格或制表符来开始每个解析器的定义。然后，随后每一行以单个制表符开头的行将用于定义该解析器。所有行都应为小写，并且对于字符串只能包含 '_'、'@' 或 '#' 字符，任何其他字符（除括号和空白字符（包括逗号）外）都将被视为错误。元解析器 unfortunately 不是为健壮性而设计的，因此请务必使用单个制表符开始每一行，在行的其他地方使用空格而非制表符进行对齐，并且不要在特定解析器的定义中插入额外的空行。

例如，从视觉上看，解析器定义可能如下所示：
```
type_name
	line 1
	line 2
	line 3
	...
```

#### 定义如何处理值

本节首先描述当你事先知道文本标签 `yyy` 将是什么时，解析形如 `yyy = zzz` 的值的选项。请注意，`,` 将被视为空白字符，因此可以自由用于格式化目的。要解析这样的值，请在解析器定义中添加如下一行：
```
	yyy, value, type, handler_type
```
或如下一行：
```
	yyy, value, type, handler_type (destination_name)
```
关键字 `value` 必须存在，以告诉生成器我们正在解析一个名为 `yyy` 的值，而不是一个组。`type` 可以是以下之一：`float`、`int`（对应 `int32_t`）、`uint`（对应 `uint32_t`）、`bool`、`text` 或 `date`（`date` 在撰写本文档时目前不受支持，因为我还没有需要它）。这只是对 `zzz` 进行常见转换的一种便捷方式。如果你指定 `text`，你将获得 `zzz` 的文本，并去除任何周围的引号（如果存在）。`handler_type` 必须是 `discard`、`member`、`member_fn` 或 `function` 之一（其效果如下所述）。最后，`(destination_name)` 如果存在，必须始终用括号包围。在处理 `zzz` 时，键 `yyy` 也用作成员、成员函数或自由函数的名称，除非你通过指定 destination name 来覆盖此选择（你可以使用此功能将多个值发送到同一个处理程序，例如，或避免违反 C++ 标识符规则，如果 `zzz` 恰好是保留关键字）。

对于你添加的每一行来处理一个值，你需要准备好一些东西来接受它，除非你想丢弃它，这就是为 `handler_type` 指定 `discard` 的作用。如果你为 `handler_type` 指定 `member`，则与解析器关联的类必须具有正确的 destination name 的成员，并且其类型具有可以接受解析后的 `zzz` 类型的 `operator=`。如果 `handler_type` 是 `member_fn`，则该类必须具有 destination name 的成员函数，并且能够接受以下参数：一个 `association_type` 类型的枚举、解析后的 `zzz` 的类型、一个 `error_handler` 类的对象（按引用）、一个包含读取值左侧所在行号的 `int32_t`，以及解析器的上下文（按值或按引用）（当不使用上下文时，惯例是将其类型设为 int32_t）。`association_type` 参数允许你检查该值存在的 association type，你可以使用 `error_handler` 和行号来记录该值由于任何原因而无法接受的情况。最后，`function` 的工作方式与 `member_fn` 相同，只是它调用具有指定 destination name 的自由函数，并作为额外的第一个参数传递用于解析的对象（按引用）。`function` 并没有提供太多额外的灵活性，我将尽量避免使用它。

除了处理左侧是预先已知的固定量的情况外，你还可以将值的左侧和右侧都发送到你选择的例程（假设它不匹配显式指定的左侧）。这样做需要如下一行：
```
	#any, value, type, handler_type (destination_name)
```
和往常一样，括号中的内容也可以完全省略。与显式指定左侧的情况一样，文本 `value` 是必需的，以表明此行适用于值。`type` 和 `handler_type` 也采用与之前基本相同的值和含义。`discard` 作为处理程序类型的作用与以前一样，如果你想丢弃所有未显式匹配的其他值而不生成任何错误，它会很有用。`member` 在这种情况下大多无用，因为它不传达关于值表达式左侧的任何信息。`member_fn` 和 `function` 会修改，它们将收到一个额外的 `string_view` 作为其第一个或第二个参数，分别保存值表达式的左侧。

#### 定义如何处理组

指定如何处理组的完成方式与值大致相同，但有更多选项，因为你可以将必要的行写成：
```
	yyy, group, type, handler_type (destination_name)
```
或
```
	yyy, extern, function_name, handler_type (destination_name)
```
其中 `(destination_name)` 和之前一样是可选的。这两行都将指示生成的解析器如何处理看起来像 `yyy = { ... }` 的组项目。`group` 和 `extern` 之间的区别在于，`group` 将通过调用另一个生成的解析器（通过调用与类 `type` 关联的解析器）来处理该组的内容（即构成 `...` 的项目），并传入该组的内容和当前上下文。另一方面，`extern` 将调用具有给定名称的函数，并传入三个参数：一个 `token_generator` 类型的对象（按引用）、一个 `error_handler` 类的对象（按引用），以及解析器的上下文（按值或按引用）。这在你实际使用上下文时很有用。使用 `extern` 的典型模式是简单地将其用作创建一个包含更多信息的上下文的地方，然后再调用另一个生成的解析器来处理它。

`handler_type` 和 `destination_name` 的作用与值基本相同，只是它们处理的是解析后的值右侧或函数返回值，唯一的区别是没有 `association_type` 参数，因为它对组没有意义。

**注意** `group` 和 `extern` 之间的另一个 minor difference 是，即使你为 `extern` 指定了 `discard` 处理，它仍将在该组的内容上运行，这在你倾向于立即处理数据且没有任何有用的东西可返回时很有用。是的，这使得阅读定义有点误导。

与值一样，你可以将 `yyy` 替换为 `#any` 以解析左侧的标记未预先固定的组。与值一样，这将导致一个额外的 `string_view` 参数被发送到处理程序。此外，由于 `extern` 而被调用的函数还将被传递一个包含 `=` 左侧文本的 `string_view` 作为额外的第一个参数（对于由于 `group` 而调用的自动生成解析器不可用，因为它们不能接受额外的参数）。

#### 定义如何处理自由值

要定义如何处理遇到的任何自由值，请在解析器定义中添加如下一行：
```
	#free, value, type, handler_type (destination_name)
```
`value` 必须存在，以表明这是你希望处理自由值的方式，而不是处理自由集。与常规值一样，`type` 确定文本将立即被解析为哪种值类型，`handler_type` 确定如何处理该值，`destination_name` 如果存在，则覆盖处理程序的默认名称（如果你不指定某些内容，则默认为 `free_value`）。

如果 `handler_type` 是 `member_fn`，则该类必须具有一个成员函数，能够接受以下参数：解析后的 `zzz` 的类型、一个 `error_handler` 类的对象（按引用）、一个包含读取值左侧所在行号的 `int32_t`，以及解析器的上下文（按值或按引用）。（**注意** 与标准值的区别在于没有 `association_type` 枚举）`function` 的工作方式与 `member_fn` 相同，只是它调用一个自由函数，并作为额外的第一个参数传递用于解析的对象（按引用）。

请注意，你最多只能有一个自由值处理程序。如果你需要能够接受不同类型的自由值（例如数字和文本数据），那么你必须为类型指定 `text`，然后自己决定它是哪种值。

#### 定义如何处理自由组

要定义如何处理自由组，请在解析器定义中添加如下一行：
```
	#free, group, type, handler_type (destination_name)
```
或
```
	#free, extern, function_name, handler_type (destination_name)
```
功能上，自由组的工作方式与正常组完全相同，只是它们的默认 destination name 是 `free_group`。但你真的不需要担心它；自由组非常罕见，我甚至不确定 Victoria 2 文件中是否有一个存在（是的，这些解析器的规范是从一个基于 CK2 文件的更旧规范派生而来的——别问）。

#### 重用现有解析器定义中的项目

最后，可以以某种有限的方式在一个解析器定义中重用另一个解析器定义的内容。你可以通过添加如下一行来实现：
```
	#base, name
```
其中 name 是你之前在同一文件中用于某个其他解析器定义的名称。这本质上通过“复制”先前定义的内容来工作，并且没有任何覆盖它的规定；你只能添加新项目。

### 关于以这种方式生成的解析器的简要说明

生成的解析器可以调用为 `parsers::parse_typename(token_gen, error_record, context)`，其中 `token_gen` 的类型为 `parsers::token_generator`，`error_record` 的类型为 `parsers::error_handler`，`context` 可以是任何你想要的东西，尽管惯例是在不使用上下文时传递 `0`。

`token_generator` 通过向构造函数传递两个 `char` 指针来创建，这两个指针指示应从中提取标记的文本范围（以通常的 C++ 风格，因此第二个指针指向范围结束后的下一个内存位置）。通常你应该每个文件只创建一个标记生成器。

`error_handler` 通过向构造函数传递一个包含错误应归因的文件名称的字符串来创建。要向该类型的对象添加自定义错误，只需将描述性消息（理想情况下包括存储在成员 `file_name` 中的文件名（作为 `std::string`）和行号）附加到其成员 `accumulated_errors`，其类型为 `std::string`。还要确保你的错误消息以 `\n` 结尾。惯例是假设如果 `accumulated_errors` 的长度为零，则解析文件时没有错误。目前没有办法表达警告。
