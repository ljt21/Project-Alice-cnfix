# 迈向更好的 UI 系统

## 终极目标

制作一个所见即所得（WYSWYG）工具来设计 UI，甚至包括命名描述 UI 行为的函数。在项目的 UI 文件上运行该工具应该产生两样东西：(a) 生成的 C++ 文件，包含 UI 的原型等（也就是说，尽可能多的无需手写的代码，除了定义所需 UI 行为的函数体）；(b) 一个单一文件，可能是二进制格式，编码所有可在运行时加载的 UI 信息；(c) 一个本地化文件，或者可能是本地化文件的存根。

不把一切都打包进 C++ 文件的目的是为了尽可能保留可修改性。然而，从头开始解析项目文件几乎肯定是太低效的（在 Windows 上打开小文件本身效率就很低，而且杀毒软件通常还会给文件操作增加额外开销）。因此，我们将尝试把 UI 打包进一个单一文件。这可能会让两个都想修改 UI 的模组更难兼容，但我稍后再担心这个问题。

## 自举

编写 UI 代码很烦人，而且 WYSWYG 工具本身也需要大量复杂的 UI。因此，计划是用新的 UI 系统来编写工具本身。直接这样做意味着我们必须用工具来制作工具，这是不可能的。所以我们需要一个自举过程。这意味着至少在命令行工具中编写系统的初始实现，该工具可以读取文本编码的项目文件并生成各种生成文件。然后我们将使用这些生成的文件来制作第一个正式版本的工具。一旦我们有了足够简单可用的控件和选项，就可以用简化版的工具来编辑更复杂的功能到它自己的项目文件中，从而用工具来完成它自己。这也使得工具本身成为我们开发过程中系统的便捷测试平台。

## 文本方向

我打算支持原生 RTL（从右到左）文本，方法是水平翻转 UI 布局和大多数图标。如果你一开始就预期要这样做，这是一种相当容易的变换。现在，在一个完美的世界里，我还希望支持竖排文本，即使据我所知，只有传统蒙古文是唯一仍然需要它的活语言。不幸的是，旋转布局比镜像布局复杂得多，因为通常我们不能同时旋转顶层 UI 容器的尺寸。这意味着旋转布局需要更复杂的逻辑来重新处理以适应新尺寸，而过去的经验告诉我这是一个巨大的麻烦。所以，至少在不久的将来，我不计划支持竖排文本（向传统蒙古文致歉）。

## 通用渲染

渲染将支持使用图像文件以各种方式进行通用的拉伸和纹理贴图（不过，可能不支持旋转）。它还将支持"画笔填充"，这是一种如今似乎不太常见的技术。画笔填充的工作原理是查阅调色板。调色板中的每个条目要么是一种纯色，要么是一种重复纹理（对我们而言，不过你可以想象添加渐变和其他程序性画笔）。然后你用这支"画笔"绘制一个单色遮罩来创建最终图像。（而且，就像早期的印刷技术一样，你可以用不同的遮罩和画笔进行两次或更多次 passes 来创建更复杂的图像。）就我个人而言，我喜欢将这项技术与矢量图形结合来做各种效果。实际上，在这个项目中，这对文本渲染至关重要。我们将把文本以单色方式渲染到纹理上，然后将该纹理用作遮罩，在屏幕上绘制实际文本。这使我们能够轻松切换画笔，以不同方式渲染文本的不同部分（例如，显示文本选区）。

我们可能还需要以某种方式支持裁剪遮罩来简化渲染过程。在我之前的项目中，我通过设置裁剪来实现弹出容器的 UI 元素（例如展开菜单），以便后续元素不能绘制在它们上面。我不知道这在 OpenGL 中有多容易实现，所以可能需要其他解决方案。

我们需要能够将 UI 的一部分渲染到纹理中，以便能够渲染 UI 动画。

## UI 行为

一个不能做事的 UI 毫无价值。然而，为了让事情更简单，我们希望在 UI 创建工具中说明按钮的作用，而不是必须在 C++ 中将其连接到行为（很痛苦）。

### 暴露函数

为了实现这一点，我们必须做两件事。首先，当我们加载"编译后"的 UI 文件时，需要填充函数指针以使行为生效。这部分就是生成 C++ 代码的作用。在制作项目时，我们将每个行为函数映射到一个 id，在生成的代码中，我们将用这些相同的 id 填充一个包含适当函数指针的数组。因此，加载过程将能够仅使用生成文件中的 id 来适当地填充函数指针。（为了让模组工作，显然工具中需要有一个切换选项，可以固定现有 id 并防止添加新 id，这样模组作者就不会意外破坏需要重新编译的东西。）

### 获取属性

属性将可以通过编译时类型索引和代码生成的标识符值在 UI 树中向上检索。你将能够通过值（如果未找到则获得默认构造的值）或指针（如果未找到则为 nullptr）来访问值。由于元素可以拥有哪些属性将部分由我们在运行时加载的文件控制，这可能还涉及一些生成的代码技巧。特别是，我们可能会为 UI 对象及其属性分配内存，以及一些关于这些属性是什么的可变尾部空间信息（为了安全起见，全部初始化为 0）。一个快速版本也可供元素获取自己的某个属性，而无需向上遍历树。

（注意：如何获取特定属性？每种元素类型都有一个在表中注册的 id。然后用你想要的属性索引到表中，它会告诉你该属性不可用，或者它在尾部区段中的偏移量。）

### 调用函数

同样，有时需要能够在其他 UI 元素上调用函数。这是通过上面提到的表来完成的，它允许我们从 id 转到该函数类的实现。我们将提供方法，用于在支持它的最近父级上调用函数，在所有最近的子级上调用它，以及自调用。

## 本地化与翻译

我们将使用一个总体上基于 PrintUI 项目工作的系统，但因为模组希望能够组合额外的文本，所以 somewhat 复杂。（与 UI 修改不同，后者我们可以假设一次只有一个模组想触碰。）显然，在运行时用文本键查找是完全不现实的。同时，我们需要在运行时加载文本以支持语言切换，而且我们可能不想把所有语言的所有文本都塞进场景文件中。因此，场景文件将存储创建时看到的键以及它们文本内容的槽位。启动时，我们将加载本地化文件的内容，并将每个键指向其未解析的内容。最后，我们将在 UI 中需要时解析相关文本并缓存结果。由于此文本仅在 UI 中使用，我们还可以安全地在 UI 线程中重新加载文件（以切换语言）。

我们将通过在 localization 目录（带 z，以便与 v2 目录区分开）中为每种语言和文字组合存储一个目录来进行本地化。例如，我们可能有 `localization\en-US`。在该目录中会有一个 fonts 目录和一个 text 目录。text 目录将包含一个或多个 .txt 文件，其中包含匹配特定键的文本。fonts 目录将包含若干字体文件（.otf 和 .ttf 格式）以及若干索引文件。每个索引文件将描述 UI 中引用的一个字体槽位。索引文件将形成字体的层级结构，从首先绘制字形的基础字体开始，然后按顺序命名后备字体，以在找不到字形时尝试使用。索引文件还将存储各种字体参数（包括大小，但不包括颜色），这些参数定义了给定字体槽位应该是什么样子（例如字重、是否斜体等）。

## 总结：生成的代码

- 包含从类型 id 到尺寸的映射
- 包含从类型 id 到默认构造函数和析构函数（来自 void*）的映射
- 为所有需要实现的行为函数提供原型
- 包含从行为函数 id 到函数指针的映射
- 特殊名称的映射，当创建时指向全局可查找的 UI 元素（应该以某种方式放在根中，以防我们想要支持创建 UI 集合的多个实例）

## 总结：加载编译后的 UI

- 加载所有 UI 元素类型的定义（如果我们支持从多个文件加载，可能会重新映射内部 id）
- 存储任何所需的函数指针
- 最终……实例化根元素

## 总结：加载本地化文件

- 在 keymap 中查找键 -> 将键槽内容指向未解析的文本
- 需要为 v2 本地化文件提供备用处理（事件等需要）

## 总结：库文件

- 实现事件处理的 UI 逻辑
- 实现渲染逻辑
- 实现布局
- 实现可访问性与文本服务
- 实现 UI 元素实例化函数
