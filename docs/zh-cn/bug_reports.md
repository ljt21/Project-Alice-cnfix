# 如何报告错误（并让它们得到修复）

## 实际流程

在 discord 的 `bug-reports` 论坛中发帖（首选），或者，如果你不在 discord 上，在 github 仓库中提交一个 issue。

## 提交有用的错误报告

描述你遇到的问题在很多方面来说是错误报告中最没用的部分。这是因为错误报告的目的不仅仅是提醒开发者问题存在，而是帮助他们修复问题。在某些情况下，问题本身的描述足以找出根本原因，但在大多数情况下是不够的。因此，如果你想看到问题被修复，你可能需要向我们提供一些额外信息。

你能做的最好的事情是告诉我们重现问题所需的精确步骤。换句话说，你需要给我们一个配方，当我们完全按照它执行时，能让我们亲眼看到问题。如果你能提供这样的配方，你遇到的问题*将会*被修复，而且可能相当及时。还要注意，这个配方需要从头开始。如果你的 bug 需要非常特定的游戏条件，你还需要告诉我们如何让游戏进入该条件。运行数百场游戏试图找到看到 bug 的正确条件是不可行的。

然而，有时这可能不可能。请记住，要修复问题，开发者需要能够自己重现它，所以虽然你可能不知道确切的重现步骤，但任何修复的第一步都将是开发者找出这些步骤。为了使修复更快或更有可能发生，请尝试提供尽可能多的潜在相关信息。尝试描述导致问题发生之前发生了什么，特别是任何 beforehand 发生的不寻常的事情（通过这，我不是指"有 bug"，而是指你做的或遇到的任何不属于典型游戏会话的事情）。

## （可能）不会被修复的 bug

一个足够模糊的错误报告（例如，如果你将某件事描述为"奇怪"或"不寻常"而没有进一步细节）：这是因为开发者根本不知道他们实际上在寻找什么，而且试图找到如此模糊描述的问题对任何人来说都不值得花时间。

一个基本上是对原始游戏规则或我们对这些规则所做的有意更改持不同意见的错误报告：我们愿意讨论此类问题，但它们不是 bug，不应该作为 bug 报告。

没有重现步骤或崩溃转储的崩溃报告：这类报告包含的信息太少，根本无法采取行动。请注意，虽然崩溃转储很有帮助，但如果你能找出的话，重现问题的步骤更有用。
