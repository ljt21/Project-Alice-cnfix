## 存档和场景文件格式

除了设置文件外，我们生成两种类型的文件：存档文件和场景文件。场景文件包含定义特定模组组合加载的游戏起始情况所需的信息。（每种不同的模组组合，包括不使用任何模组运行游戏，都需要自己的场景文件。）相比之下，存档文件只包含可能从场景文件给定的状态中发生变化的游戏状态信息。因此，特定事件的定义会在场景文件中找到，但不会在存档文件中找到，因为该信息在游戏过程中不会发生变化。然而，一个国家的当前威望会在场景文件和存档文件中都找到。开始新游戏只需要加载场景文件。恢复存档游戏则首先需要加载适当的场景文件，然后加载存档文件。

就兼容性而言，Data Container 库生成的数据相当稳健，因为可以加载存储时对象或属性更多或更少的数据，甚至属性类型已更改的数据。然而，生活在 Data Container 之外的数据存储得就没那么稳健了。当这类数据发生变化时，我们将不得不增加文件版本号，并直接拒绝版本错误的文件。希望这种情况不会经常发生，也希望我们能把大部分修改限制在 Data Container 内部。

### 如何手动序列化 Data Container 之外的信息

如果你向游戏状态添加了需要在单个会话之外持久化的信息，而该信息*没有*存储在 Data Container 中（并且没有打上适当的 `save` 或 `scenario` 标签），那么你需要将它添加到为场景或存档游戏存储的数据中。无论哪种情况，保存和加载数据的函数都在 `serialization.cpp` 中实现。我们只支持两类对象的简单序列化：(1) 可以用 `memcpy` 简单移动的对象（因此不需要在构造函数或析构函数中执行操作，且不包含任何指针），或 (2) 这类对象的 `vector`。对于情况 (1)，你需要添加以下三项：
- 在计算存储所有数据所需大小（以字节为单位）的函数中添加 `sz += sizeof(state.my_new_variable);`
- 在将我们要序列化的字节写出的函数中某处添加 `ptr_in = memcpy_serialize(ptr_in, state.my_new_variable);`，以及
- 在从原始字节读取数据的函数中，以与你在前一个函数的相同相对位置添加 `ptr_in = memcpy_deserialize(ptr_in, state.my_new_variable);`。（让生活轻松一点：只需在两种情况下都在函数末尾添加你的新项目，这样可以轻易地确保它们以相同的相对位置存储和读取。）

对于情况 (2)，你需要改为执行以下操作：
- 在计算存储所有数据所需大小（以字节为单位）的函数中添加 `sz += serialize_size(state.my_new_vector);`
- 在将我们要序列化的字节写出的函数中某处添加 `ptr_in = serialize(ptr_in, state.my_new_vector);`，以及
- 在从原始字节读取数据的函数中，以与你在前一个函数的相同相对位置添加 `ptr_in = deserialize(ptr_in, state.my_new_vector);`。

一旦你更改了任何手动数据，你可能应该增加格式版本号，这样我们就不会意外尝试加载任何可能还在周围的旧文件。通过在 `serialization.hpp` 中增加 `M` 或 `N` 值来实现：
```
constexpr inline uint32_t save_file_version = M;
constexpr inline uint32_t scenario_file_version = N + save_file_version;
```
（在实际文件中 M 和 N 都是整数。）如果你更改了存档文件中的内容，增加 M；如果你只更改了影响场景的内容，增加 N。

### 场景文件

场景文件由三个主要部分组成。首先，有一个简短的头部部分。头部的确切内容尚未决定，但目标是在这里存储允许启动器显示场景信息而无需完全加载它的信息。我们生成的任何校验和也可能放在头部中。文件的第二部分包含作为二进制 blob 的场景数据。这包括地图和文本数据，但只包含各种其他资源的文件名，而不是它们的完整内容（因此图形不会也打包进场景文件中）。最后，第三部分包含本质上是一个存档游戏的内容，代表世界的初始状态。虽然开始新游戏时需要这些信息，但当场景文件与现有存档文件一起加载时，不需要加载它。

#### 二进制布局

#### 头部

```
4 字节   |   （小端序）整数，包含头部部分的长度（以字节为单位，不包括这前四个字节）
4 字节   |   版本号 -- 单调递增；版本号错误的文件将不会被加载
N 字节   |   头部部分的剩余内容（待定）
```

#### 场景

```
4 字节   |   （小端序）整数，包含此部分的长度（以字节为单位，不包括这前 8 个字节）
4 字节   |   （小端序）整数，包含此部分解压缩后的大小（以字节为单位）
N 字节   |   此部分的压缩内容（使用 zlib）
```

#### 初始游戏状态

```
4 字节   |   （小端序）整数，包含此部分的长度（以字节为单位，不包括这前 8 个字节）
4 字节   |   （小端序）整数，包含此部分解压缩后的大小（以字节为单位）
N 字节   |   此部分的压缩内容（使用 zlib）
```

### 存档文件

与场景文件一样，存档文件包含一个较短的头部，后跟一个较长的包含其大部分数据的部分。除了包含解释存档内容所需的详细信息（例如玩家国家和游戏日期）外，头部还指明了加载存档所需的场景文件（可能包括其校验和）。

#### 头部

```
4 字节   |   （小端序）整数，包含头部部分的长度（以字节为单位，不包括这前四个字节）
4 字节   |   版本号 -- 单调递增；版本号错误的文件将不会被加载
N 字节   |   头部部分的剩余内容（待定）
```

#### 游戏状态

游戏状态部分与场景文件中找到的初始游戏状态部分完全相同（意味着可以使用相同的函数来加载和保存它）。

```
4 字节   |   （小端序）整数，包含此部分的长度（以字节为单位，不包括这前 8 个字节）
4 字节   |   （小端序）整数，包含此部分解压缩后的大小（以字节为单位）
N 字节   |   此部分的压缩内容（使用 zlib）
```
